# Issue: `Shroud` Trait Should Be `unsafe trait` (Soundness Issue)

## Summary

The `Shroud<T>` trait is declared as a **safe trait**, but its contract requires implementors to uphold safety invariants that the compiler cannot check. A malicious or incorrect `Shroud` implementation can cause **undefined behavior** (use-after-free, invalid pointer dereference) without ever writing the `unsafe` keyword. This is a **soundness hole**.

## Why This Is an Issue

Per the [Rust Nomicon's guidance on unsafe traits](https://doc.rust-lang.org/nomicon/safe-unsafe-meaning.html), a trait should be `unsafe` when:
- Its implementations must uphold invariants that the compiler cannot check.
- Violating those invariants can lead to undefined behavior in safe code.

The `Shroud<T>` trait has this exact signature:

```rust
// phylactery/src/shroud.rs, line 10-12
pub trait Shroud<T: ?Sized> {
    fn shroud(from: NonNull<T>) -> NonNull<Self>;
}
```

This trait is used in `Soul::bind()` to produce the raw pointer stored inside a `Lich`:

```rust
// phylactery/src/soul.rs, line 94-100
pub fn bind<S: Shroud<T> + ?Sized>(self: Pin<&Self>) -> Lich<S> {
    increment(&self.count);
    Lich {
        count: self.count_ptr(),
        value: S::shroud(self.value_ptr()),
    }
}
```

The `Lich` then dereferences this pointer in its `data_ref()` method:

```rust
// phylactery/src/lich.rs, line 89-93
fn data_ref(&self) -> &T {
    // Safety: the pointers are valid for the lifetime of `self`; guaranteed by the
    // reference count.
    unsafe { self.value.as_ref() }
}
```

**The problem**: Since `Shroud` is a safe trait, any user can implement it without writing `unsafe`:

```rust
use core::ptr::NonNull;
use phylactery::Shroud;

// No `unsafe` keyword needed - this compiles fine!
impl Shroud<i32> for dyn core::fmt::Debug {
    fn shroud(_from: NonNull<i32>) -> NonNull<Self> {
        // Return a completely bogus pointer - causes UB when dereferenced
        unsafe { NonNull::new_unchecked(0x1 as *mut dyn core::fmt::Debug) }
    }
}
```

When `Soul::bind()` calls this implementation, the resulting `Lich` will contain an invalid pointer. Any subsequent `Deref` access (`*lich`) will dereference this invalid pointer, causing **undefined behavior**.

## Impact

- **Severity**: High (soundness hole)
- **Affected code**: All users of the `Shroud` trait, which is the core mechanism for creating `Lich` instances.
- **Required for UB**: The user must implement `Shroud` incorrectly, but they can do so without writing `unsafe`.

## Reproduction

In practice, the existing Shroud implementations (both macro-generated and hand-written in `shroud.rs`) are correct. The issue is that the **type system does not prevent** incorrect implementations.

A user implementing `Shroud` for their own trait would not be warned by the compiler that they need to maintain the invariant that the returned `NonNull<Self>` must point to the same allocation as the input `NonNull<T>`.

## Proposed Fix

1. Mark the trait as `unsafe trait`:

```rust
/// # Safety
///
/// Implementors must guarantee that `shroud(from)` returns a `NonNull<Self>`
/// that:
/// - Points to the same allocation as `from`.
/// - Has valid metadata (vtable pointer) for `Self`.
/// - The returned pointer is valid for reads as long as `from` is valid.
pub unsafe trait Shroud<T: ?Sized> {
    fn shroud(from: NonNull<T>) -> NonNull<Self>;
}
```

2. Update all implementations to use `unsafe impl`:
   - In `phylactery/src/shroud.rs`: All `impl` blocks generated by `shroud_ty!` and `shroud_fn!` macros.
   - In `phylactery_macro/src/lib.rs`: The generated `impl` blocks from the `#[shroud]` attribute macro.

3. Update documentation to explain the safety requirements.

## Files to Modify

- `phylactery/src/shroud.rs` (lines 10-12): Add `unsafe` to trait declaration and all `impl` blocks.
- `phylactery/src/shroud.rs` (lines 27-65): Add `unsafe` to all `shroud_ty!` generated impls.
- `phylactery/src/shroud.rs` (lines 93-121): Add `unsafe` to all `shroud_fn!` generated impls.
- `phylactery_macro/src/lib.rs` (lines 64-94): Add `unsafe` to all macro-generated impls.
- `phylactery/tests/shroud.rs`: Verify tests still compile.

## Notes

This is a **semver-breaking change** because:
- Existing user implementations of `Shroud` will need to add `unsafe impl`.
- The trait bound `S: Shroud<T>` in `bind()` will now require `unsafe` implementations.
- This would necessitate a major version bump (or at least a minor version bump with a migration guide).

However, it is the correct thing to do for soundness. The `Send` and `Sync` traits in `std` are `unsafe` for exactly the same reason.
